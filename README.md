# Бот "График дежурств" — MAX messenger

Бот для управления графиком дежурств: автоматический круг ротации, ручные правки,
уведомления по расписанию, опрос по итогам смены, автогенерация протокола,
статистика, разграничение доступа.

## Установка

```
pip install -r requirements.txt
cp config.example.json config.json   # заполнить реальными данными
cp .env.example .env                 # вписать MAX_BOT_TOKEN
python bot_max.py
```

На хостинге без файловой системы (Railway/Render/bothost и т.п.) вместо
`config.json` можно передать содержимое файла целиком в переменную окружения
`CONFIG_JSON`.

**Важно:** сразу впишите свой числовой ID (узнать через `/myid`) в
`config.json`/`CONFIG_JSON` → `admin_ids`. Без этого доступ в админку
(`/admin login`) не переживает рестарт бота — на некоторых хостингах
файловая система между рестартами не сохраняется.

**Ручные дежурства** (поверх автоматического круга ротации) задаются в
`config.json`/`CONFIG_JSON` → `duty_schedule` — списком объектов
`{"date": "дд.мм.гггг", "employees": [...], "phones": [...], "is_pair": true/false}`.
В коде (`bot_max.py`) этих данных нет намеренно: `config.json` не попадает в
git (см. `.gitignore`), а любые реальные ФИО и телефоны, зашитые прямо в
исходник, оказались бы в публичном репозитории.

## Функции для всех пользователей

| Команда / кнопка | Что делает |
|---|---|
| `/start` (или просто открыть чат с ботом) | Регистрация; для привязки к сотруднику — ввести в чат последние 4+ цифры своего номера телефона. Ручной выбор ФИО из списка — только для админа (иначе любой мог бы привязаться к чужому имени без знания номера) |
| 📋 Полный график | Показывает график дежурств на 12 недель вперёд |
| 👤 Моё дежурство | Ближайшие 3 дежурства привязанного сотрудника |
| 📄 Скачать протокол | Присылает загруженный админом шаблон протокола |
| ❓ Частые вопросы / 📝 Инструкция | Статичные справочные тексты |
| 🔄 Изменить профиль | Перепривязать аккаунт — снова ввести последние цифры телефона |
| 📊 Статистика дежурств / `/stats` | Кол-во дежурств и % участия по сотрудникам — неделя / месяц / всё время |
| `/contact` | Показывает ваш профиль (сотрудник + телефон) |

**Важно:** статистика (`/stats`) считается только по сменам, начиная с момента
появления этой функции — прошлые дежурства бот задним числом не восстановит,
история пустая до первого реального фотоотчёта.

## Автоматические уведомления (по расписанию, всем пользователям)

| Когда | Что |
|---|---|
| Среда 18:00 | Кто дежурит в эту субботу |
| Четверг 15:00 | Дежурному на ближайшую субботу — **пустой бланк `protocol_template.docx`**, чтобы успеть распечатать заранее |
| Пятница 18:00 | Напоминание о завтрашнем дежурстве |
| Суббота 08:00 (настраивается в `config.json` → `survey.send_hour/minute`) | **Запрос фотоотчёта** — только дежурному(ым) в личку |
| Суббота 10:00 | Итоги дежурства всем |

### Фотоотчёт о дежурстве
Вместо текстового опроса дежурному приходит просьба прислать **фото протокола**
следующим сообщением. Бот ждёт именно фото (любой другой ответ — переспрашивает).
Полученное фото сохраняется и попадает в список "Список дежурств" в админке.

**Хранится максимум 5 последних фото** — при поступлении 6-го самое старое
удаляется с диска (но факт "прислал/не прислал" остаётся в статистике навсегда).

Автогенерация `.docx`-протокола (заполнение `protocol_template.docx` датой)
больше не привязана к фотоотчёту — доступна отдельно через `/test_protocol`.

## Админ-функции

Вход: `/admin AdminDuty admin123` (логин/пароль в `config.json`) — либо
автоматически, если ваш user_id указан в `config.json` → `admin_ids`
(тогда `/admin` вообще не нужен и подтверждение ниже не требуется).

**Двухфакторный вход.** Правильный пароль сам по себе прав не даёт — это
только первый фактор. После него всем уже действующим админам (тем, кто
залогинен через `/admin`, либо перечислен в `admin_ids`) приходит уведомление
«Попытка входа в админ-панель» с именем и ID того, кто пытается войти, и
кнопками «✅ Принять» / «❌ Отклонить». Права выдаются только после
подтверждения кем-то из них. Пользователи из `admin_ids` это подтверждение
не проходят — они и так доверены конфигом и входят сразу.

| Раздел | Возможности |
|---|---|
| 📅 Управление графиком | Добавить/удалить ручное дежурство поверх круга, `/set_schedule` |
| 👥 Управление сотрудниками | Добавить/удалить сотрудника, изменить телефон (`/set_phone`), список |
| 📁 Управление файлами | Загрузить протокол-шаблон, закрепить его в чате, удалить, проверить |
| 📋 Список дежурств | Последние 5 смен: кто дежурил / дата / фото протокола (или "не отправил доказательства") |
| 📊 Статистика (админская) | Технические цифры: пользователи, автопривязка, ближайшее дежурство |

### Как загрузить протокол (важно!)
В MAX нельзя прикрепить текст-подпись к файлу в одном сообщении, поэтому
загрузка работает **в два шага**: сначала нажимаете кнопку
(«Загрузить протокол» или «Прикрепить протокол»), бот переходит в режим
ожидания, и **следующий отправленный файл** (без подписи) обрабатывается
автоматически.

### Тестовые команды (только для админов)
| Команда | Что делает |
|---|---|
| `/test_wednesday` | Форсирует уведомление среды |
| `/test_friday` | Форсирует уведомление пятницы |
| `/test_saturday` | Форсирует уведомление субботы |
| `/test_report` | **Форсирует запрос фотоотчёта** в любой день недели (для теста без ожидания субботы) |
| `/test_protocol_reminder` | **Форсирует отправку пустого бланка** на ближайшую субботу (обычно уходит по четвергам в 15:00) |
| `/test_protocol` | **Сразу генерирует и присылает .docx протокол** (только дата) — для быстрой проверки формата файла, в статистику не попадает |
| `/reset_shifts confirm` | **Полностью очищает историю дежурств** (статистику, список дежурств, все сохранённые фото) — удобно снести накопленные тестовые записи перед реальным использованием |
| `/myid` | Показывает ваш числовой ID — впишите его в `config.json`/`CONFIG_JSON` → `admin_ids`, чтобы получить права админа без `/admin login` и не терять их при рестарте бота |
| `/time` | Показывает текущее время сервера, следующее уведомление и версию задеплоенного кода |
| `/users` | Список всех пользователей и их статус |

## Защита от посторонних (выключена по умолчанию)

Если включить `whitelist_mode: true` в `config.json` — все новые пользователи
не смогут ничем пользоваться, кроме кнопки «🔓 Запросить доступ». Запрос
уходит всем админам с кнопками «Разрешить» / «Отклонить».

## Логирование

- Обычные логи — в консоль/stdout, как раньше.
- `audit.log` (новый файл) — кто/когда/что сделал: входы в админку, изменения
  графика и сотрудников, загрузка протоколов, ответы на опрос, решения по
  запросам доступа.

## Чек-лист для тестирования

### Базовое
- [ ] `/start` от нового пользователя → ввести последние 4 цифры своего номера в чат → главное меню
- [ ] Ввести цифры, которые не совпадают ни с одним номером → бот пишет «не найдено», без кнопки списка ФИО (для не-админа)
- [ ] `/start` от админа (залогинен или в `admin_ids`) → видна кнопка «📋 Выбрать ФИО из списка», ручная привязка работает
- [ ] Не-админ не может привязаться к чужому ФИО, даже зная payload `emp_<Имя>` — callback отклоняется
- [ ] Открыть чат с ботом заново (не /start) → меню должно появиться само (BotStarted)
- [ ] 📋 Полный график, 👤 Моё дежурство, ❓ Вопросы, 📝 Инструкция — открываются

### Статистика
- [ ] `/stats` → три кнопки периода → при пустой истории пишет «Нет завершённых дежурств»
- [ ] После теста фотоотчёта (см. ниже) — числа дежурств должны появиться

### Фотоотчёт, бланк-напоминание, список дежурств
- [ ] `/test_report` от админа → дежурному(ым) в личку приходит просьба прислать фото
- [ ] Отправить текст вместо фото → бот переспрашивает («нужно именно фото»)
- [ ] Отправить фото → бот пишет «Фотоотчёт получен»
- [ ] Админ-панель → 📋 Список дежурств → эта смена должна быть в списке с фото
- [ ] `/test_protocol_reminder` → дежурному на ближайшую субботу приходит пустой бланк `.docx`
- [ ] Прислать 6 фотоотчётов подряд (через 6 `/test_report`) → в 📋 Список дежурств должно остаться не больше 5 файлов, самый старый — с пометкой «файл удалён (лимит 5 файлов)»

### Админка
- [ ] `/admin AdminDuty admin123` от аккаунта **не из `admin_ids`** → приходит «Пароль верный, ожидайте подтверждения», а уже действующим админам — уведомление с кнопками «Принять/Отклонить»
- [ ] Нажать «Принять» → у запросившего появляется доступ в админ-панель, приходит уведомление о подтверждении
- [ ] Нажать «Отклонить» → доступ не выдаётся, приходит уведомление об отказе
- [ ] `/admin AdminDuty admin123` от аккаунта **из `admin_ids`** → вход сразу, без подтверждения
- [ ] Управление графиком → добавить дежурство → проверить в 📋 Полный график
- [ ] Управление сотрудниками → добавить/удалить/изменить телефон
- [ ] Управление файлами → «Загрузить протокол» → прислать `.docx` **без подписи** → должен сохраниться
- [ ] «Прикрепить протокол» → прислать файл → должен закрепиться в чате
- [ ] `/set_phone`, `/set_schedule`, `/contact` — команды должны вести туда же, что и кнопки

### Доступ и безопасность
- [ ] Если включите `whitelist_mode: true` — новый пользователь должен увидеть «Доступ ограничен» и кнопку запроса
- [ ] Запрос доступа → админу приходит уведомление с кнопками → «Разрешить» открывает доступ
- [ ] Проверить файл `audit.log` рядом с ботом — там должны появляться строки по каждому действию выше

### Сессия админа
- [ ] Перезапустить бота → без повторного `/admin login` попытка загрузить файл должна показать «Сессия администратора не активна» (если вашего ID нет в `admin_ids`)
- [ ] Если ваш ID есть в `config.json` → `admin_ids` — админ-функции должны работать сразу после рестарта, без `/admin login`
